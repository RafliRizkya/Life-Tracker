## How to work with Rafli (learned, not written elsewhere)

- **Move fast, don't stall on routine approval.** Rafli explicitly said to assume approval by default rather than asking. Exception: when a request conflicts with a standing `CLAUDE.md` rule (e.g. "don't change nav structure") — name the conflict and proposed resolution explicitly, get a real answer for *that*, but don't add approval checkpoints beyond it.
- **Ponytail (lazy-code) mode is the default posture** for implementation work in this repo — smallest correct diff, reuse before adding, no speculative abstractions.
- **Prefer real data over placeholders when it exists.** The Career Journey redesign replaced 8 fictional seed milestones with 13 parsed from Rafli's actual resume — that's the expected default, not a one-off.
- **Verification is part of "done," not optional.** Production build + a Playwright smoke pass (click through the actual flow, screenshot desktop/mobile 390px/dark mode, check console errors) before calling a UI change complete.
- **Never commit automatically.** Draft the commit message, stage nothing without being asked. Rafli has committed doc-sync work himself before (see log below) — don't assume silence means "go ahead and push."
- **Docs are a living deliverable.** After a feature-level change, `CLAUDE.md`/`PRD.md`/`SRS.md` need updating in the same pass — Rafli caught this being skipped once already (see 2026-07-18 entry below) and had to ask for it explicitly. Don't wait to be asked next time.

### 2026-07-19 — Cross-device sync (Supabase)
Rafli caught that data diverged between his laptop and phone and asked directly for Supabase sync — persistence was `localStorage`-only; the only existing Supabase usage was the paused WhatsApp webhook. Found Supabase already fully configured (real credentials in both `backend/.env` and `frontend/.env.local`) but `supabase/migrations/0001_initial_schema.sql`'s 11 relational tables are unusable from the browser: their RLS policies gate on `auth.uid()`, and this app has no login system, so `auth.uid()` is always null and every policy denies access. Presented Rafli a real trade-off via `AskUserQuestion` before building (per `CLAUDE.md`'s "explain plan, wait before architectural changes"): single JSONB state-blob table (small build, matches existing `persistableSlice()` shape) vs. full per-entity relational CRUD (correct long-term shape, multi-day build). He picked the blob, and picked "sync everything including private reflections" over carving those out (his reasoning: private means never shown to *other people*, not never leaves the device). Shipped `supabase/migrations/0003_app_state_sync.sql` (one row per user, RLS enabled with zero policies — service-role-only by construction, no auth needed), `frontend/src/app/api/sync/route.js` (GET pull / POST push via the service-role client), and reworked `hydrate()`/`persist()`/`reseed()` in `store.js` for local-first render + background reconcile + 800ms-debounced push, last-write-wins by timestamp. Verified with Playwright using two isolated browser contexts (no shared storage, i.e. genuinely simulating two devices) — add on "device A" reached Supabase and appeared on a brand-new "device B" context within ~1.5s; delete round-tripped the same way; confirmed clean via direct SQL after. Full detail: `docs/features/supabase-sync.md`.

### 2026-07-19 — Fix: transactions silently failing to persist
Source: `docs/prompt/fix-crud-db.md` (reported as "transactions don't save to the database" — there is no traditional DB in this architecture, so the real question was why localStorage writes were being lost). Root cause found in `frontend/src/lib/store.js`: `saveToStorage()` wrapped `localStorage.setItem` in a try/catch that swallowed every error (comment: "quota exceeded — ignore"). `persist()`'s own try/catch never saw the failure, so `saveStatus` reported `"saved"` even when the write never happened — any add/delete looked instant in the UI but reverted on refresh (private-mode browsers throw `QuotaExceededError` on every `setItem`, which is the likely real-world trigger). Fix: removed the inner catch so the error propagates to `persist()`, which now also pushes a deduped `"save-failed"` warning notification (bell-icon badge, tone `warning`) alongside the existing topbar "Save failed" indicator — reused the existing notification system rather than adding a toast library. Guarded `hydrate()`'s first-visit branch (which also calls the now-throwing `saveToStorage`) by routing it through `persist()` instead of calling storage directly. No schema change, no new dependency. Verified: production build clean, dev server serves `/finance` 200 with no console errors.

### 2026-07-18 — Dev server "feels stuck" navigation fix
Root cause: Next.js dev server's `onDemandEntries` default (5-page buffer, 60s idle disposal) — with ~9 routes in this app, navigating between sections evicted and recompiled routes repeatedly, each recompile costing 2–4s and feeling like a freeze. Fixed in `frontend/next.config.js` (`onDemandEntries: { maxInactiveAge: 1hr, pagesBufferLength: 15 }`, dev-only, no effect on production builds). Measured before/after with Playwright: before, revisiting an evicted route cost 2.4–3.7s indefinitely through a session; after, every route compiles once then stays sub-second (0.13–0.35s) for the rest of the session. First-visit-per-session compile cost is unavoidable in dev mode — for zero compile lag even on first visit, the real fix is running `npm run build && npm run start` instead of `npm run dev`.
