## How to work with Rafli (learned, not written elsewhere)

- **Move fast, don't stall on routine approval.** Rafli explicitly said to assume approval by default rather than asking. Exception: when a request conflicts with a standing `CLAUDE.md` rule (e.g. "don't change nav structure") — name the conflict and proposed resolution explicitly, get a real answer for *that*, but don't add approval checkpoints beyond it.
- **Ponytail (lazy-code) mode is the default posture** for implementation work in this repo — smallest correct diff, reuse before adding, no speculative abstractions.
- **Prefer real data over placeholders when it exists.** The Career Journey redesign replaced 8 fictional seed milestones with 13 parsed from Rafli's actual resume — that's the expected default, not a one-off.
- **Verification is part of "done," not optional.** Production build + a Playwright smoke pass (click through the actual flow, screenshot desktop/mobile 390px/dark mode, check console errors) before calling a UI change complete.
- **Never commit automatically.** Draft the commit message, stage nothing without being asked. Rafli has committed doc-sync work himself before (see log below) — don't assume silence means "go ahead and push."
- **Docs are a living deliverable.** After a feature-level change, `CLAUDE.md`/`PRD.md`/`SRS.md` need updating in the same pass — Rafli caught this being skipped once already (see 2026-07-18 entry below) and had to ask for it explicitly. Don't wait to be asked next time.

### 2026-07-18 — Dev server "feels stuck" navigation fix
Root cause: Next.js dev server's `onDemandEntries` default (5-page buffer, 60s idle disposal) — with ~9 routes in this app, navigating between sections evicted and recompiled routes repeatedly, each recompile costing 2–4s and feeling like a freeze. Fixed in `frontend/next.config.js` (`onDemandEntries: { maxInactiveAge: 1hr, pagesBufferLength: 15 }`, dev-only, no effect on production builds). Measured before/after with Playwright: before, revisiting an evicted route cost 2.4–3.7s indefinitely through a session; after, every route compiles once then stays sub-second (0.13–0.35s) for the rest of the session. First-visit-per-session compile cost is unavoidable in dev mode — for zero compile lag even on first visit, the real fix is running `npm run build && npm run start` instead of `npm run dev`.
